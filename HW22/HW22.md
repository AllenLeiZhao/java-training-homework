## 1. Point-to-point vs Publish-subscribe

**Point-to-point (queue)**: a producer puts a message on a queue, and **exactly one** consumer takes it. Multiple consumers on the same queue *compete* for messages; a message is gone once it is acknowledged. Classic examples: a job queue, SQS, a RabbitMQ work queue.

**Publish-subscribe (topic)**: a producer publishes to a topic, and **every subscriber** gets its own copy. Subscribers don't compete — they each see the full stream. Classic examples: SNS, RabbitMQ fanout exchange, Kafka topics read by different consumer groups.

| | Point-to-point | Publish-subscribe |
|---|---|---|
| Delivery | Each message → one consumer | Each message → all subscribers |
| Best at | Distributing *work* | Broadcasting *events* |
| Scaling | Add consumers → linear throughput | Add subscribers → new functionality |
| Coupling | Producer knows there's "a worker" | Producer knows nothing about consumers |

**Point-to-point pros**: natural load balancing (add workers to drain faster); each unit of work is processed once; simple backpressure — the queue absorbs bursts; easy to reason about ("N jobs in, N jobs done").
**Cons**: no fan-out — if a second system also needs the data, you must change the producer to write to a second queue; consuming is destructive, so there is no replay for a consumer that had a bug.

**Pub-sub pros**: total decoupling — new subscribers (analytics, audit, cache invalidation) can be added without touching the producer; one event can drive many downstream behaviors; fits event-driven architecture.
**Cons**: harder to guarantee that every subscriber actually processed every message (a slow or offline subscriber is *its own* problem); every subscriber receives the full volume whether it needs it or not; more moving parts to monitor.

The interview punchline: **Kafka merges both models.** Within one consumer group, partitions are divided among members → point-to-point work sharing. Across consumer groups, every group gets the whole stream → pub-sub. That duality is a big part of why Kafka won.

## 2. Why a message system at all?

Without one, services call each other synchronously (REST), which couples them in time, availability, and capacity. A message system in the middle buys:

1. **Decoupling** — the producer doesn't know who consumes, or even how many consumers exist. Teams deploy independently.
2. **Asynchrony** — the caller returns immediately ("order accepted") instead of waiting for email, inventory, billing to all finish inline.
3. **Peak shaving / buffering** — a flash-sale burst of 50k requests/sec lands in the broker; consumers drain it at their sustainable 5k/sec instead of falling over. The queue trades latency for survival.
4. **Resilience** — if the consumer is down for 10 minutes, messages wait; nothing is lost and the producer never noticed.
5. **Fan-out & replay** (pub-sub/Kafka) — the same order event feeds fulfillment today and a new fraud model next quarter, which can even re-read last week's events.
6. **Ordering per key** — Kafka guarantees order within a partition, so "all events for order 42" stay in sequence.

Concrete tie to my EMS project: creating an employee currently writes to PostgreSQL synchronously. If we needed to also send a welcome email, sync to LDAP, and update a search index, doing that inline would make `POST /api/employees` slow and fragile; publishing one `EmployeeCreated` event and letting three consumers react is the message-system answer.

## 3. Kafka architecture

```
 Producers ──▶ Topic "orders" (partitioned, replicated across brokers)
                                                            Consumer group "billing"
              Broker 1          Broker 2          Broker 3    ┌──────────────┐
             ┌─────────┐      ┌─────────┐      ┌─────────┐    │ C1 ← P0      │
             │ P0 (L)  │      │ P1 (L)  │      │ P2 (L)  │    │ C2 ← P1, P2  │
             │ P1 (F)  │      │ P2 (F)  │      │ P0 (F)  │    └──────────────┘
             └─────────┘      └─────────┘      └─────────┘   Consumer group "analytics"
                                                              (gets the full stream too)
             L = leader, F = follower replica; metadata & controller: KRaft quorum
```

The pieces:

- **Topic** — a named stream of records, split into **partitions**.
- **Partition** — an append-only, ordered log on disk. The unit of parallelism and of ordering: order is guaranteed *within* a partition only. A record's key decides its partition (`hash(key) % partitions`), so all events for one key stay ordered.
- **Offset** — a record's position in its partition. Consumers track "where am I" as (group, topic, partition) → offset, stored in the internal `__consumer_offsets` topic. Because consuming is just moving a pointer, reads are non-destructive → replay is free.
- **Broker** — one Kafka server; owns some partitions. A cluster is many brokers.
- **Replication** — each partition has one **leader** (all reads/writes) and N−1 **followers** on other brokers. Followers that keep up form the **ISR** (in-sync replica set). `acks=all` + `min.insync.replicas=2` means a write is only acknowledged once safely on ≥2 replicas; if the leader's broker dies, the controller promotes an in-sync follower.
- **Producer** — appends records; chooses partition by key (or round-robin/sticky if keyless); batches and compresses for throughput.
- **Consumer group** — the scaling unit for reading. Each partition is assigned to exactly one consumer *within* a group (work sharing); different groups each get everything (pub-sub). A group with more consumers than partitions leaves the extras idle.
- **Controller / metadata** — historically ZooKeeper; modern Kafka uses **KRaft** (built-in Raft quorum), one less system to operate.
- **Storage** — each partition is a series of **segment files**; Kafka writes sequentially and serves reads via the OS page cache and zero-copy, which is why a "disk-based" system can be this fast.

## 4. Messages are accumulating — consumers can't keep up. What do I do?

First **measure**: consumer lag (`kafka-consumer-groups.sh --describe`, or Burrow/Grafana) tells me which topic/partition lags and whether lag is growing or just spiky. Then, in order:

1. **Find the real bottleneck.** Usually the consumer isn't slow at *reading* — it is slow at what it does per message (a synchronous DB write, a REST call). Fix the per-message work first: batch the DB writes, cache lookups, parallelize downstream I/O.
2. **Scale consumers horizontally** — add consumers to the group, up to the number of partitions. That is the designed elasticity.
3. **Out of headroom? Add partitions** (consumers can't exceed partition count). Caveat: `hash(key) % partitions` changes, so key→partition mapping shifts; ordering across the boundary is disturbed once — plan it, don't do it in panic.
4. **Tune the consumer**: bigger `fetch.min.bytes` / `max.poll.records` for batch efficiency; make sure processing isn't inside one giant transaction; commit offsets in batches; consider pipelining (poll thread + worker pool) as long as offset commits stay after processing.
5. **Degrade gracefully if it's a flood**: park non-critical event types to a separate topic/DLQ, apply sampling for low-value events, or (business permitting) skip-ahead — reset offsets past data that is now worthless, explicitly and with sign-off.
6. **Producer side / long term**: rate-limit or batch producers, compress, and re-do the capacity math (below) so partitions and consumers match projected peak, not average.

The trap answer to avoid: "just restart the consumer" or "just add partitions" — without finding why per-message processing is slow, more partitions only spread the same bottleneck (usually the database) thinner.

## 5. How does Kafka deal with expired data?

Kafka doesn't delete on consumption; it deletes by **retention policy**, per topic (`cleanup.policy`):

- **`delete`** (default): data older than `retention.ms` (default 7 days) or beyond `retention.bytes` is removed. The granularity is a whole **segment file** — when a segment's newest record is past retention, the log cleaner deletes the file. This is why expiry is cheap: no per-record scanning, just unlinking old segments.
- **`compact`**: log compaction keeps **the latest value per key** forever and discards older values for the same key. Perfect for changelog/state topics (e.g. `__consumer_offsets` itself, or a "current employee record" topic) — you keep a full snapshot without unbounded growth. A record with a null value (*tombstone*) marks the key for deletion.
- **`compact,delete`**: both — compact per key *and* drop everything older than retention.

Two consequences worth saying in an interview: consumers are not notified — if a slow consumer's next offset has already been deleted, it hits `OffsetOutOfRange` and follows `auto.offset.reset` (`earliest`/`latest`); and retention is a *contract with downstream teams* — "you have 7 days to consume or replay" is a design decision, not a default to leave untouched.

## 6. Data volume in Kafka

Kafka's design goal is that volume is a *provisioning* question, not an architectural limit — production clusters run from GB/day to PB/day (LinkedIn famously reported trillions of messages per day). What the question is really testing is whether I can **estimate**:

```
daily volume   = msgs/sec × avg msg size × 86,400
disk needed    = daily volume × retention days × replication factor  (+ ~30% headroom)
```

Worked example — say our EMS-style order system:

- 2,000 msgs/sec average, 1 KB per message
- → 2 MB/sec ≈ **170 GB/day** of raw data
- retention 7 days, replication factor 3 → 170 × 7 × 3 ≈ **3.6 TB**, plus headroom ≈ **4.5–5 TB** across the cluster

Per-broker rule-of-thumb sanity checks: a plain broker on decent disks comfortably sustains tens of MB/s and low-thousands of partitions; if the math says one broker needs 500 MB/s or 20k partitions, the answer is "more brokers," not "hope."

## 7. How to calculate the partition number

Partitions = parallelism, so size them from **throughput targets**, measured not guessed:

```
P = max( T_target / T_producer_per_partition , T_target / T_consumer_per_partition )
```

- `T_target` — peak throughput the topic must handle (design for peak × growth, not average).
- `T_producer_per_partition` — what one producer can push into one partition (measure; often ~10s of MB/s).
- `T_consumer_per_partition` — what one consumer achieves per partition **including its processing** (often the smaller number by far — e.g. 5 MB/s if each message triggers a DB write).

Example: target 100 MB/s peak; producer does 20 MB/s per partition, a consumer only 5 MB/s → `max(100/20, 100/5) = max(5, 20) = 20 partitions`, then round up for growth → 24–30.

Adjustments beyond the formula:

- **Consumer count**: partitions ≥ the max consumers you ever want in one group.
- **Key ordering**: all events for one key hit one partition — a single hot key can't be parallelized by adding partitions.
- **Overhead ceiling**: every partition costs open files, replication traffic, and rebalance/leader-election time. Thousands per broker is fine; hundreds of thousands per cluster is an operations problem. Don't create 1,000 partitions for a 50 msg/sec topic "just in case."
- **Changing later is awkward** (key remapping, §4), so provision for ~1–2 years of growth up front.

## 8. AWS SQS vs RabbitMQ vs Kafka

| | **AWS SQS** | **RabbitMQ** | **Kafka** |
|---|---|---|---|
| Model | Managed point-to-point queue | Smart broker: exchanges route to queues (direct/topic/fanout) | Distributed, replicated *log*; consumer groups = queue + pub-sub |
| Message after consumption | Deleted | Deleted (ack) | **Retained** until retention expires → replay |
| Ordering | Standard: best-effort; FIFO queues: ordered but ~300–3k msg/s | Per-queue | Per-partition (scales with partitions) |
| Throughput | High (managed, virtually unlimited for Standard) | ~10k–50k msg/s typical | Very high — hundreds of MB/s per broker, scales linearly |
| Delivery | At-least-once (Standard), exactly-once-ish on FIFO | At-least-once (ack/nack, DLX) | At-least-once; exactly-once with idempotent producer + transactions |
| Routing logic | None (queue per purpose) | **Richest** — routing keys, headers, priorities, per-message TTL, delays | None in broker — dumb broker, smart consumers |
| Ops burden | Zero (serverless, pay-per-use) | You run it (or CloudAMQP) | Highest (cluster, partitions, monitoring) — or managed MSK/Confluent |
| Sweet spot | Decouple AWS microservices, background jobs, cheap & simple | Complex task routing, RPC patterns, per-message priorities, moderate scale | Event streaming, high volume, replay, multiple independent readers, stream processing |

How I'd choose: **inside AWS and just need to decouple two services → SQS** (zero ops wins). **Complex routing rules / task semantics / priorities → RabbitMQ.** **High-volume event stream that several teams consume and may need to replay → Kafka.** It's events-as-history (Kafka) vs commands-in-transit (SQS/RabbitMQ).

## 9. AWS SNS vs SQS

Complementary, not competing:

| | **SNS** | **SQS** |
|---|---|---|
| Model | Pub-sub **topic** | Point-to-point **queue** |
| Delivery | **Push** to all subscribers (Lambda, HTTP, email, SMS, SQS) | Consumers **poll** |
| If consumer is down | Retries, then message is **lost** (per retry policy) | Message **waits** (up to 14 days) |
| Persistence | No storage — delivery attempt only | Durable until consumed/expired |
| One message goes to | All subscribers | One consumer |

The standard production pattern is **SNS → SQS fan-out**: publish once to an SNS topic; subscribe one SQS queue per consuming service (with subscription **filter policies** so each queue only gets relevant messages). Each service then gets SQS's durability, its own retry pace, and a DLQ, while the producer keeps SNS's one-to-many decoupling. Order events: SNS topic `order-events` → queues `billing`, `shipping`, `analytics` — billing being down for an hour affects nobody else, and no order is lost.

Rule of thumb: **SNS answers "who else needs to know?", SQS answers "who will do the work?"** — and gluing them together is usually the right answer.
