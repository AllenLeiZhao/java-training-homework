## 1. Spring Security in my EMS project

I secured my **employee-department** project (https://github.com/AllenLeiZhao/employee-department) with Spring Security 7 (Spring Boot 4). Design in one picture:

```
                 ┌──────────────────────────────────────────────┐
 HTTP Basic ─────┤                                              │
 Form login ─────┤  SecurityFilterChain                         │──▶ SecurityContext
 Google OAuth2 ──┤  (one chain, three ways to authenticate)     │    (username + ROLE_*)
                 └──────────────────────────────────────────────┘
 Authorization rules (URL level):
   /name/aggregation              -> permitAll   (classmate chain + Jenkins verify stage)
   GET  /api/**                   -> ROLE_USER or ROLE_ADMIN
   POST/PUT/DELETE /api/**        -> ROLE_ADMIN
   anything else                  -> authenticated
 Authorization rules (method level):
   EmployeeController.deleteEmployee -> @PreAuthorize("hasRole('ADMIN')")  (defense in depth)
```

**Authentication (who are you?)** — three entry points that all end in the same place:
- **HTTP Basic** for API clients (curl / Postman / tests).
- **Form login** for browsers, served from a small custom `/login` page that shows error/logout feedback.
- **Google OAuth 2.0 / OIDC login** ("Login with Google" button on the same page).

**Authorization (what may you do?)** — accounts live in a new `app_users` table (`AppUser` entity: username, email, BCrypt password hash or `null` for Google accounts, role, provider). A `DbUserDetailsService` loads local users; passwords are stored **only as BCrypt hashes** (`PasswordEncoder = BCryptPasswordEncoder`). A startup seeder creates one `admin` and one `user` account idempotently, with passwords overridable via environment variables.

Key implementation files:

| File | Responsibility |
|---|---|
| `security/SecurityConfig.java` | Filter chain, URL rules, the three login mechanisms, 401-instead-of-redirect for `/api/**` |
| `security/DbUserDetailsService.java` | Local account lookup; refuses password login for OAuth-provisioned accounts |
| `security/GoogleOidcUserService.java` | Maps a Google identity onto our `app_users` row; auto-provisions first-time logins with role USER |
| `security/AppUserSeeder.java` | Idempotent seed of admin/user with BCrypt hashes |
| `entity/AppUser.java`, `repository/AppUserRepository.java` | Account persistence |
| `controller/LoginController.java` | Self-contained `/login` page: username/password form + Google button, error/logout notices |
| `controller/MeController.java` | `/api/me` introspection: shows name, authorities, auth type |

Two decisions worth defending in an interview:
- **CSRF is disabled** because this is a JSON API consumed with explicit credentials (Basic auth) rather than cookie-only browser sessions; the protection for state-changing endpoints comes from role rules. For a cookie-based browser app I would leave CSRF on.
- `/name/aggregation` stays **public on purpose**: it is the node other classmates' services call in the aggregation chain, and the Jenkins pipeline health-checks it anonymously.

## 2. Google OAuth 2.0 registration and login

**Registration steps (Google Cloud Console):**
1. https://console.cloud.google.com → create project `employee-department`.
2. *APIs & Services → OAuth consent screen*: External, app name, my email; scopes `openid`, `email`, `profile`; add myself as test user.
3. *APIs & Services → Credentials → Create Credentials → OAuth client ID*: type **Web application**; authorized redirect URI:
   - `http://localhost:8080/login/oauth2/code/google` (local)
4. Copy the generated **Client ID** and **Client Secret**.

**Why Google login is demonstrated on localhost only:** Google's OAuth 2.0 policy requires redirect URIs to (a) use **HTTPS** and (b) be a **domain name, not a raw IP** — `localhost` is the single exemption. Our EC2 deployment is `http://3.128.166.190:8080`, which violates both rules, so Google rejects it with a "doesn't comply with security standards" error before the consent screen. Making it work in the cloud would require a domain + TLS termination (e.g. a free DuckDNS subdomain + a Caddy/Nginx reverse proxy with a Let's Encrypt certificate, plus `server.forward-headers-strategy=framework` so Spring generates an `https://` redirect_uri behind the proxy). That is deliberately out of scope for this homework; on EC2 the app still authenticates via HTTP Basic and form login, and the OAuth flow is verified end-to-end locally.

The credentials are **never committed**. They are injected as environment variables:

```properties
spring.security.oauth2.client.registration.google.client-id=${GOOGLE_CLIENT_ID:dummy-client-id}
spring.security.oauth2.client.registration.google.client-secret=${GOOGLE_CLIENT_SECRET:dummy-client-secret}
spring.security.oauth2.client.registration.google.scope=openid,profile,email
```

On EC2 they live in `/home/ec2-user/employee-service.env` (chmod 600), which `docker run --env-file` picks up — the Jenkinsfile creates a placeholder on first deploy and I fill in the real values on the host, so secrets stay out of git, out of the image, and out of Jenkins logs.

**Login flow (authorization code + PKCE, verified locally):** `GET /oauth2/authorization/google` → 302 to `accounts.google.com/o/oauth2/v2/auth?...&code_challenge=...&code_challenge_method=S256` → user consents → Google redirects back to `/login/oauth2/code/google?code=...` → Spring exchanges the code for tokens, validates the ID token, and calls my `GoogleOidcUserService`, which:
1. rejects Google accounts that expose no email;
2. finds the `app_users` row by email, or **auto-provisions** one (role USER, provider GOOGLE, no password);
3. returns an `OidcUser` whose authorities come **from our database**, not from Google — Google authenticates, my application authorizes.

## 5. Security concepts, in my own words

### Encryption vs Hashing vs Encoding

Three different intents, often confused because all three "scramble" data:

| | Purpose | Reversible? | Key? | Examples |
|---|---|---|---|---|
| **Encoding** | Representation/transport | Yes, by anyone | No | Base64, URL-encoding, UTF-8 |
| **Encryption** | Confidentiality | Yes, with the key | Yes | AES (symmetric), RSA (asymmetric) |
| **Hashing** | Integrity / fingerprint | No (one-way) | No (salt ≠ key) | SHA-256, BCrypt |

Encoding is not security at all — Base64 "protects" nothing; anyone can decode it. Encryption protects secrecy and is meant to be reversed by whoever holds the key. Hashing is a one-way fingerprint: same input → same digest, but you cannot go back. That's exactly why passwords are **hashed, never encrypted**: the server doesn't need the original password, it only needs to verify a match — and if the database leaks, there is no key that unlocks all passwords at once. For passwords specifically you use a *slow, salted* hash (BCrypt/Argon2) rather than fast SHA-256, so brute-forcing leaked hashes is expensive. In my project: BCrypt for passwords, TLS (encryption) for transport, Base64 only inside the Basic-auth header and JWT segments (pure encoding).

### Authentication vs Authorization

- **Authentication (authn)** = *who are you?* Proving identity: password, Google login, certificate. Fails with **401 Unauthorized**.
- **Authorization (authz)** = *what may you do?* Checking permissions of an already-known identity. Fails with **403 Forbidden**.

They always run in that order. In my EMS app the line is visible: `user:user123` authenticates fine (no 401) but `POST /api/employees` still returns **403** because USER lacks the ADMIN role. With Google login the split is even clearer — Google does authentication, but authorization is decided entirely by the role stored in my `app_users` table.

### HTTPS, certificates, key rotation

**HTTPS** is HTTP inside a TLS tunnel, giving confidentiality (nobody can read it), integrity (nobody can alter it), and server authenticity (you're talking to the real server). The TLS handshake in one breath: client and server agree on cipher suites, the server presents its **certificate**, they run a key exchange (ECDHE, giving forward secrecy) to derive a shared symmetric session key, and everything after is fast symmetric encryption (AES-GCM) — asymmetric crypto only bootstraps trust.

A **certificate** binds a public key to a domain name, signed by a Certificate Authority. The browser walks the chain (leaf → intermediate → trusted root) and checks domain match, validity window, and revocation. Self-signed certs fail this because no trusted third party vouches for them.

**Key rotation** = replacing keys/certificates on a schedule, not only after an incident. It limits the blast radius (a stolen key is only useful until rotation), and practicing rotation means that when a real leak happens, replacing the key is routine instead of an outage. Applies to TLS certs (Let's Encrypt automates ~90-day renewal), JWT signing keys (publish old + new via a JWKS endpoint, verify with `kid`, then retire the old one), and DB credentials/API keys. Rule of thumb: if rotating a key scares the team, that's exactly why it must be rotated regularly.

### JWT token structure

A JWT is three Base64Url segments joined by dots: `header.payload.signature`.

- **Header** — metadata: `{"alg":"RS256","typ":"JWT","kid":"key-1"}` (which algorithm/key signs it).
- **Payload** — the claims: registered (`iss` issuer, `sub` subject, `aud` audience, `exp`/`iat`/`nbf` times) plus custom ones like `roles`.
- **Signature** — `sign(base64url(header) + "." + base64url(payload), key)`. With HS256 the same shared secret signs and verifies; with RS256 the private key signs and anyone with the public key verifies — better for microservices.

Two things I'd stress: the first two segments are **encoded, not encrypted** — anyone can read them, so no secrets in the payload; the signature only guarantees *integrity* (tamper → verification fails). And JWTs are stateless, which means they can't be easily revoked before `exp` — so keep lifetimes short and pair with refresh tokens. Google's ID token in my OAuth flow is exactly such a JWT, and Spring validates its signature, issuer and audience before trusting it.

### OAuth 2.0 and OIDC

**OAuth 2.0** is a *delegated authorization* framework: a user lets application A access their resources at service B without giving A their B password. Roles: resource owner (user), client (my EMS app), authorization server (Google), resource server (Google APIs). The flow I use is **authorization code + PKCE**: redirect to Google → user consents → Google returns a one-time `code` via the registered redirect URI → my backend exchanges code + client secret for tokens. The access token never passes through the browser URL, and PKCE (`code_challenge`/`code_verifier`) stops a stolen code from being replayed by a different client.

**OIDC (OpenID Connect)** is a thin *identity* layer on top of OAuth 2.0. Plain OAuth answers "may I access this?"; it says nothing standard about *who the user is*. OIDC adds the `openid` scope, an **ID token** (a JWT with `sub`, `email`, signed by Google) and the `/userinfo` endpoint — turning OAuth into a login protocol ("Sign in with Google"). In my app: OAuth 2.0 mechanics move the tokens, the OIDC ID token proves identity, and my `GoogleOidcUserService` maps that identity to local roles.

### LDAP

LDAP (Lightweight Directory Access Protocol) is a protocol for querying a hierarchical **directory service** — think of a company phone book that also stores accounts, groups and machines (Microsoft **Active Directory** is the classic implementation). Entries form a tree of DNs like `uid=azhao,ou=engineering,dc=company,dc=com`. Enterprises use it as the *single source of truth* for identity: an application performs a **bind** (authentication attempt) with the user's DN and password, and reads group memberships to derive roles — so one directory backs SSO for email, VPN, Git, and internal apps, and disabling one account revokes everything. Spring Security supports it almost drop-in (`spring-security-ldap`); my `DbUserDetailsService` could be swapped for an LDAP-backed `UserDetailsService` without touching the authorization rules — same authn/authz split, different identity store.

### SQL injection and prepared statements

SQL injection = user input being concatenated into a SQL string so that *data becomes code*:

```java
// vulnerable
String sql = "SELECT * FROM app_users WHERE username = '" + name + "'";
// name = "' OR '1'='1' --"  → returns every user, bypasses login
```

**Prepared statements** fix it by separating the query *structure* from the *values*: the SQL template with `?` placeholders is parsed and planned first, then values are bound as pure data — the driver never re-interprets them as SQL, so `' OR '1'='1` is just a weird literal string that matches nobody.

```java
PreparedStatement ps = conn.prepareStatement("SELECT * FROM app_users WHERE username = ?");
ps.setString(1, name);
```

In my project I never write raw SQL: Spring Data JPA derived queries (`findByUsername(...)`) and JPQL with named parameters all compile down to prepared statements with bound parameters. Defense in depth on top: Bean Validation on DTOs, least-privilege DB account, and no error-message leakage of SQL details. Prepared statements also get a performance bonus — the execution plan is reused across calls.
