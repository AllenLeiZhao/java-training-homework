## 1. Why spring boots? pros and cons
Spring Boot makes it much faster to build a Spring application by eliminating boilerplate configuration, but the trade-off is that it can feel like a black box if you don't understand what's happening underneath.

The main advantages are: auto-configuration, which means Spring Boot automatically sets up the application based on the dependencies you add, so you don't have to write XML or Java config from scratch; embedded server, which means you don't need to deploy to an external Tomcat, you just run the jar file; and starter dependencies, which bundle commonly used libraries together so you only need to add one dependency instead of five. The downside is that because so much is done for you automatically, it can be hard to debug when something goes wrong, and the application startup time and memory footprint are heavier compared to a lightweight framework.

## 2. How to start a spring boot project from scratch?
I use Spring Initializr at start.spring.io to generate the project structure, then import it into IntelliJ and start building.

I go to start.spring.io, choose Maven as the build tool and Java as the language, set the group and artifact name, pick the Java version, and then add the dependencies I need, like Spring Web, Spring Data JPA, and a database driver. I download the zip, open it in IntelliJ, and Maven automatically downloads all the dependencies. Then I create the entity, repository, service, and controller in the three-tier structure, configure the database connection in application.properties, and run the main class to start the application.

## 3. @Controller vs @RestController
@RestController is a combination of @Controller and @ResponseBody, so every method automatically serializes the return value to JSON and writes it directly to the response body.

@Controller is the original annotation for the web layer. By default it expects you to return a view name, like a Thymeleaf template. If you want to return data instead of a view, you have to add @ResponseBody to every method. @RestController does that automatically for every method in the class, so you don't have to repeat yourself. In practice, if you're building a REST API, you always use @RestController. @Controller is only needed when you're returning HTML views.

## 4. @PathVariable vs @RequestParam
@PathVariable extracts a value from the URL path itself, while @RequestParam extracts a value from the query string after the question mark.

For example, if the URL is slash users slash 5, you use @PathVariable to get the 5, because it's part of the path and it identifies a specific resource. If the URL is slash users question mark age equals 25, you use @RequestParam to get the 25, because it's a filter or optional parameter in the query string. The rule of thumb is: use @PathVariable to identify a resource, use @RequestParam to filter or sort results.

## 5. @RequestBody vs @ResponseBody
@RequestBody reads the incoming request body and deserializes it into a Java object, while @ResponseBody takes the return value of a method and serializes it into the response body, usually as JSON.

@RequestBody is used on a method parameter, typically for POST and PUT requests where the client sends data. Spring uses Jackson to automatically convert the JSON payload into the Java object you specified. @ResponseBody is used on a method or a class to tell Spring to write the return value directly to the HTTP response instead of looking for a view. When you use @RestController, @ResponseBody is already applied to every method automatically.

## 6. how to use GetMapping, PutMapping, PostMapping, DeleteMapping, RequestMapping?
These annotations map HTTP methods to controller methods. GetMapping, PostMapping, PutMapping, and DeleteMapping are shortcuts for the four main CRUD operations, while RequestMapping is the generic version that can handle any HTTP method.

GetMapping is for reading data, PostMapping is for creating, PutMapping is for updating, and DeleteMapping is for deleting. You put them on a method and pass in the URL path. RequestMapping can be used at the class level to set a base path that all methods in that controller share, for example slash api slash v1 slash users, and then each method adds its own sub-path on top of that. You can also use RequestMapping at the method level with a method attribute to specify GET, POST, or others, but the shortcut annotations are cleaner and more readable.

## 7. What is Spring Actuator?
Spring Actuator is a monitoring tool that exposes built-in HTTP endpoints to check the health, metrics, and internal state of a running application without stopping it.

After adding the actuator dependency and enabling the endpoints in application.properties, you can access slash actuator slash health to see if the application and its dependencies like the database are up, slash actuator slash metrics to see memory usage and request counts, and slash actuator slash loggers to change the log level at runtime without restarting. In production I'd use it alongside Prometheus and Grafana — Actuator exposes the data, Prometheus collects it, and Grafana visualizes it as dashboards. One important note: in production you should only expose the endpoints you actually need, because some of them like env can leak sensitive configuration.

## 8. how to achieve async in spring boot application
Spring Boot supports async processing with the @Async annotation, which runs a method in a separate thread so the caller doesn't have to wait for it to finish.

To use it, you first enable async support by adding @EnableAsync to a configuration class. Then you add @Async to any method you want to run asynchronously. When that method is called, Spring runs it in a thread from a thread pool and returns immediately to the caller. The method can return void if you don't need the result, or it can return a CompletableFuture if the caller needs to get the result later. A common use case is sending an email or pushing a notification — you don't want the user to wait for that, so you run it asynchronously in the background.

## 9. how does spring handle exception
Spring handles exceptions centrally using @RestControllerAdvice, which lets you define global exception handlers in one place instead of repeating try-catch blocks in every controller.

Inside a class annotated with @RestControllerAdvice, you write methods annotated with @ExceptionHandler, each one handling a specific exception type. When that exception is thrown anywhere in the application, Spring automatically routes it to the right handler method, which returns a clean error response with the appropriate HTTP status code and message. For example, if a resource is not found you'd return 404, if input validation fails you'd return 400. This keeps the controllers clean and makes error handling consistent across the entire application.

## 10. how does spring validate data
Spring validates incoming data using the Bean Validation API, specifically by adding constraint annotations to the entity fields and @Valid on the controller method parameter.

You annotate fields in your model with constraints like @NotNull to reject null values, @NotBlank to reject empty strings, @Size to enforce length limits, @Min and @Max for number ranges, and @Email for email format. Then in the controller you add @Valid before the @RequestBody parameter. If any constraint is violated, Spring automatically throws a MethodArgumentNotValidException and returns a 400 Bad Request response. You can combine this with @RestControllerAdvice to customize the error message that gets sent back to the client.

## 11. how does spring do the log
Spring Boot comes with Logback as the default logging framework, and you use it through the SLF4J interface, which is a logging facade that lets you swap out the underlying framework without changing your code.

You get a logger by calling LoggerFactory.getLogger in your class, and then you log at different levels: debug for detailed diagnostic info, info for general application flow, warn for something unexpected but not breaking, and error for failures. The log level is configured in application.properties — in production you typically set it to info or warn so you don't flood the logs with debug output. You can also change the log level at runtime through Spring Actuator without restarting the application.

## 12. cache hit vs cache miss
A cache hit means the requested data was found in the cache and returned immediately. A cache miss means it wasn't there, so the application had to go to the database to fetch it.

The whole point of caching is to avoid expensive database queries by storing frequently accessed data in a fast in-memory store. When a request comes in, the application checks the cache first. If it's a hit, you get a fast response. If it's a miss, you query the database, return the result, and also store it in the cache so the next request for the same data will be a hit. The key trade-off is consistency — cached data can become stale if the database is updated, so you need a cache invalidation strategy, like setting an expiration time or evicting the cache when the data changes.

## 13. do some research on Redis and have a basic understanding
Redis is an open-source, in-memory data store that is most commonly used as a cache, a message broker, or a session store, because it's extremely fast — all data lives in memory so reads and writes happen in microseconds.

Redis stores data as key-value pairs and supports several data structures beyond simple strings, like lists, sets, sorted sets, and hash maps. In a Spring Boot application, Redis is commonly used as the caching layer — when you query the database, you store the result in Redis with an expiration time, and future requests hit Redis instead of the database. It's also used for storing user sessions, rate limiting, and as a pub-sub message broker. The main trade-off is that since data lives in memory, it's more expensive than disk-based storage, and data can be lost if the server crashes unless you configure persistence.

## 14. @RestControllerAdvice instead of @ControllerAdvice
@RestControllerAdvice is a combination of @ControllerAdvice and @ResponseBody, so every exception handler method automatically serializes the return value to JSON. @ControllerAdvice alone is for applications that return views.

@ControllerAdvice is the general-purpose global exception handler. It works for both MVC applications that return HTML views and REST APIs. But if you're building a REST API and you use @ControllerAdvice, you have to add @ResponseBody to every handler method manually to make it return JSON. @RestControllerAdvice does that automatically for every method, just like how @RestController relates to @Controller. So for a REST API, you always use @RestControllerAdvice — it's cleaner and you don't have to repeat yourself.
