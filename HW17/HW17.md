## 1. Docker, Dockerfile, Docker Image, Docker Container
 
**Docker** is a containerization platform. Its core value is "build once, run anywhere" — it packages an application together with all of its dependencies (runtime, libraries, environment variables, config) so it runs identically across machines, eliminating the "works on my machine" problem.
 
**Dockerfile** is a text script that defines *how to build an image*. Common instructions:
- `FROM` — base image
- `COPY` / `ADD` — copy files into the image
- `RUN` — execute a command at build time (e.g. `mvn package`)
- `EXPOSE` — declare a port
- `CMD` / `ENTRYPOINT` — the command run when the container starts
It is the "recipe" for the build.
 
**Docker Image** is a read-only template — the artifact produced by building a Dockerfile. It uses a **layered** structure: each instruction creates a layer, layers are cached and reusable, which makes rebuilds fast.
 
**Docker Container** is a **running instance** of an image, with a thin read-write layer on top of the read-only image layers.
 
> OOP analogy: **Dockerfile = source code, Image = Class, Container = the object you `new`**. One image can run many isolated containers.

## 2. Docker vs VM
 
The key difference is the **level of virtualization**: a VM virtualizes hardware; Docker virtualizes the operating system.
 
| | Virtual Machine (VM) | Docker Container |
|---|---|---|
| Isolation level | Virtualizes hardware via a Hypervisor; each VM has a **full Guest OS** | Shares the host kernel; process-level isolation |
| Resource footprint | Heavy (GB-scale) | Light (MB-scale) |
| Startup speed | Slow (minutes — must boot a full OS) | Fast (seconds) |
| Isolation strength | Strong (full OS-level isolation) | Relatively weaker (shared kernel) |
| Density per host | A handful to a dozen | Dozens to hundreds |
 
**Summary:** VMs isolate more completely but are heavy; Docker is lightweight and fast to start, ideal for large-scale microservice deployment, with slightly weaker isolation. In practice the two are often combined — run Docker containers *inside* a VM (e.g. EC2).

## 3. How to Use Docker in a Real Project
 
**Consistent local environment** — run dependencies like a PostgreSQL database in a container so every team member has an identical setup, no manual installs.
 
**Package the application** — write a Dockerfile that builds the jar (`mvn package`), uses a base image like `eclipse-temurin:21-jre`, and starts the app with `java -jar app.jar`. The whole app plus JDK becomes one image.
 
**Orchestrate multiple services with docker-compose** — a single `docker-compose.yml` can bring up the app + PostgreSQL + Redis + RabbitMQ together with one `docker compose up`. Extremely convenient for local microservice integration.
 
**Integrate with CI/CD** — on code push, CI (Jenkins / GitHub Actions) automatically builds the image, pushes it to a registry (e.g. ECR), and deploys to ECS / Kubernetes.
 
**Microservice deployment** — one image per microservice, orchestrated by Kubernetes or ECS for scheduling, scaling, health checks, and rolling updates.

## 4. AWS Core Services
 
Each described as *what it is* + *how to use it in a project*.
 
**EC2 (Elastic Compute Cloud)** — virtual servers (IaaS). You control the OS, install your environment, and run your apps. Use it for services that run long-term and need full control of the environment. Most flexible, but highest operational overhead.
 
**ECS (Elastic Container Service)** — AWS's native container orchestration service; manages deployment, scaling, and health checks for Docker containers. Two launch types: **EC2 mode** (containers run on your EC2 instances) and **Fargate** (serverless — no servers to manage). Use it to deploy a cluster of Spring Boot microservices.
 
**ECR (Elastic Container Registry)** — a private Docker image registry, the AWS equivalent of Docker Hub. CI builds an image and pushes it to ECR; ECS / EKS pulls from ECR to deploy.
 
**RDS (Relational Database Service)** — managed relational database supporting MySQL / PostgreSQL / Oracle / SQL Server, etc. AWS handles backups, patching, replication, and automatic failover. Use it for structured data that needs **strong consistency and transactions**, such as orders and users.
 
**DocumentDB** — a managed document database compatible with the MongoDB protocol. Good for flexible, semi-structured data such as product details or reviews.
 
**DynamoDB** — fully managed NoSQL key-value / document database with **single-digit millisecond latency and near-unlimited horizontal scaling**. Ideal for high-concurrency, flexible-schema data such as shopping carts, sessions, and user activity.
 
**Lambda** — serverless function compute: **event-triggered, billed per invocation, auto-scaling, no servers to manage**. Good for lightweight logic such as post-order async processing, thumbnail generation, and scheduled jobs.
 
**API Gateway** — a managed API gateway handling routing, authentication/authorization, throttling, and caching. Commonly used as the single client entry point, fronting Lambda or ECS services.
 
**Kinesis** — real-time streaming data platform for ingesting and processing large-scale streams (clickstreams, logs, IoT data). Use it for real-time recommendations, real-time analytics, and event/telemetry ingestion.
 
**IAM (Identity and Access Management)** — controls *who can perform which actions on which resources*. Core concepts: User, Group, Role, Policy. The **principle of least privilege** is a high-frequency interview point.
 
**SNS / SQS** — two decoupling messaging services, often asked together:
- **SQS (Simple Queue Service)** — a message **queue**, point-to-point; each message is processed by one consumer. Used for load leveling (peak shaving) and async decoupling (e.g. drop a message on the queue after an order and process it later).
- **SNS (Simple Notification Service)** — publish/subscribe (pub-sub); one message is **broadcast** to many subscribers.
- Classic combo — **SNS + SQS fan-out**: SNS publishes once, multiple SQS queues each receive a copy, and different services consume independently.

## 5. Design: Online Shopping System (AWS only)
 
### Architecture (layered)
 
```
                     ┌─────────────────────────┐
                     │   Web / Mobile Client    │
                     └────────────┬─────────────┘
                                  │
                     ┌────────────▼─────────────┐
                     │   CloudFront (CDN)        │  ← static assets / images via S3
                     └────────────┬─────────────┘
                                  │
                     ┌────────────▼─────────────┐
                     │   API Gateway             │  ← auth (Cognito / IAM), throttling, routing
                     └────────────┬─────────────┘
                                  │
   ┌──────────────────────────────▼──────────────────────────────┐
   │   Microservices — ECS Fargate   (images pulled from ECR)     │
   │   [ User svc ] [ Product svc ] [ Order svc ] [ Payment svc ] │
   └──────────────────────────────┬──────────────────────────────┘
                                  │
   ┌──────────────────────────────▼──────────────────────────────┐
   │   Data layer                                                 │
   │   [ RDS ]      [ DynamoDB ]   [ ElastiCache ]   [ S3 ]       │
   │   orders/users  cart/session   hot-data cache   product imgs │
   └──────────────────────────────┬──────────────────────────────┘
                                  │
   ┌──────────────────────────────▼──────────────────────────────┐
   │   Async & analytics                                          │
   │   [ SNS + SQS ]       [ Lambda ]          [ Kinesis ]        │
   │   order events        email / invoice     clickstream        │
   └──────────────────────────────────────────────────────────────┘
```
 
### Request flow (walking through one order)
 
1. The client request hits **CloudFront** first — product images and front-end static assets are returned from edge nodes (backed by **S3**), reducing latency and backend load.
2. Dynamic requests (browse, checkout) enter **API Gateway**, which handles authentication (validating login state via Cognito, with IAM controlling service-to-service permissions), throttling, and **routing** to the backend.
3. Requests reach the microservice cluster on **ECS Fargate**, split by domain into User / Product / Order / Payment services — each deployed and scaled independently, with images pulled from **ECR**.
4. Data is stored by characteristic:
   - **RDS** — strongly consistent, transactional data (orders, users)
   - **DynamoDB** — high-concurrency, flexible-schema data (cart, sessions)
   - **ElastiCache** — hot data (homepage, popular products) cached to protect the database
   - **S3** — product images
5. After a successful order, the user is not blocked: the Order service publishes an "order created" event to **SNS**, which **fans out** to multiple **SQS** queues — deduct inventory, generate invoice, send notification — each consumed independently for decoupling and peak shaving. Lightweight steps (email, invoice generation) are handled by **Lambda**.
6. User browsing/click behavior is ingested in real time by **Kinesis** and fed downstream for real-time recommendations and analytics.
### Core design principles
 
- **Split into microservices by domain**
- **Choose storage by data characteristics**
- **Use message queues for async decoupling**
