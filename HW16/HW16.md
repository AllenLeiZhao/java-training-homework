### 1. What are the use cases for the following design patterns in the Banking, E-Commerce, and Health Insurance Industries?

#### Singleton
* **Banking**: An **Interest Calculation Engine Configuration Manager** that centralizes global base rates to ensure ledger consistency.
* **E-Commerce**: A shared **Payment Gateway Connection Pool** (e.g., Stripe/PayPal API clients) to optimize resource reuse under high concurrent load.
* **Health Insurance**: A global, read-only cache for standard **ICD-10/11 Disease Code Diagnostics** shared across all claim processes.

#### Factory
* **Banking**: An **Account Statement Parser Factory** that dynamically instantiates specialized parsers based on multi-channel sources (ATM, Mobile, Web).
* **E-Commerce**: A **Promotion & Discount Strategy Factory** that creates custom calculators depending on the coupon type (percentage off, fixed deduction).
* **Health Insurance**: An **Insurance Product Strategy Selector** that initializes specific claim verification logic based on policy types (Commercial, Corporate, or Underage).

#### Builder
* **Banking**: A **Mortgage Loan Application Builder** used to construct multi-layered applications (credit history, assets, guarantors) step-by-step.
* **E-Commerce**: A complex **Order Object Builder** that assembles item lists, shipping addresses, invoice parameters, and cross-border customs data.
* **Health Insurance**: An **Insurance Claim Record Builder** designed to aggregate a patient's medical history, invoices, and deductibles via an approval workflow.

#### Proxy
* **Banking**: A **Risk & Compliance Auditing Proxy** wrapping transfer methods to detect money laundering and commit distributed transactions.
* **E-Commerce**: A **Product Details Caching & Lazy-Loading Proxy** intercepting database calls to serve traffic directly from Redis and delay image fetching.
* **Health Insurance**: A **HIPAA Compliance Data Masking Proxy** that checks roles and hides Protected Health Information (PHI) like SSNs from unauthorized viewers.

### 2. What is Microservice Architecture?
A distributed architectural style where a single large application is decomposed into a collection of **small, autonomous, and loosely coupled services**. Each service models a single business capability, manages its own private database, and communicates via lightweight protocols (e.g., HTTP REST or gRPC).

---

### 3. Why Microservice Architecture?
* **Fault Isolation**: If one service fails (e.g., the review section), the rest of the application (e.g., ordering and payment) continues to function.
* **Independent Scaling**: High-demand services can be scaled out independently without wasting resources on idling modules.
* **Decoupled Development**: Teams can work on separate deployment lifecycles and freely choose the best technology stack for their specific service.

---

### 4. Microservice Main Components
* **Service Discovery**: A dynamic telephone directory maps network locations to changing service instances.
* **Config Server**: A centralized repository managing externalized configuration files across all deployment environments.
* **Client-Side Load Balancer**: A mechanism built into client services to distribute load across discovered target servers.
* **API Gateway**: A single entryway routing incoming traffic, handling security enforcement, rate-limiting, and metric logging.
* **Circuit Breaker**: A safety switch that blocks failing network calls and runs a backup fallback method to prevent cascading failures.
* **Distributed Tracing**: Injecting a unique trace ID into requests to monitor end-to-end execution across services.
* **Centralized Logging**: Shuffling log data from scattered nodes into a single indexing database for fast searching.

### 5. Looking up the following terms, know what they are and how to use.

#### Config Server
* **Concept**: Centralizes properties files in a remote Git repo, allowing services to pull updates dynamically at runtime without a complete rebuild.
* **How to use**: An external server runs `@EnableConfigServer`. Clients locate it by specifying bootstrap configurations:
    ```properties
    spring.application.name=order-service
    spring.cloud.config.uri=http://localhost:8888
    spring.cloud.config.profile=dev
    ```

#### Service Discovery: Eureka, Consul, Zookeeper...
* **Concept**: Tracks the changing network IPs and ports of operational service instances, ensuring smooth peer-to-peer networking.
* **How to use**: Enable a registry server using `@EnableEurekaServer`. Clients join the registry map automatically with a simple annotation:
    ```java
    @SpringBootApplication
    @EnableDiscoveryClient
    public class OrderApplication { ... }
    ```

#### Zipkin and Sleuth: tracing system
* **Concept**: `Sleuth` injects uniform tracking IDs into logs, while `Zipkin` parses this data into intuitive Gantt charts to track network bottlenecks.
* **How to use**: Standard log formatting automatically captures trace metadata, which is sent over to Zipkin's dashboard for visualization:
    ```text
    2026-06-19 09:15:30 [order-service,5b7f921e0a811cf3,a811cf34f5e71,true] INFO ...
    ```

#### Ribbon: client side load balancer
* **Concept**: Resides in the calling client's memory, applying smart algorithms (like Round-Robin) to route tasks across target instances.
* **How to use**: Attached directly to your HTTP request builder bean. It translates symbolic cluster names into precise IP targets:
    ```java
    @Bean @LoadBalanced
    public RestTemplate restTemplate() { return new RestTemplate(); }
    
    // Usage: restTemplate.getForObject("http://USER-SERVICE/users/1", String.class);
    ```

#### ELK: kibana, Logstash, ElasticSearch
* **Concept**: `Logstash` gathers real-time text streams from server files, `Elasticsearch` tokenizes and indexes data for swift retrieval, and `Kibana` serves as a visual graph dashboard.
* **How to use**: Ship files through standard append appenders to Logstash. Use Kibana to query error traces instantly across multiple server stacks.

#### Circuit Breaker: fall back (Hystrix -> Resilience4j)
* **Concept**: Triangulates network failures and intercepts calls once a fault rate passes threshold limits, instantly executing a local fallback method instead.
* **How to use**: Anchor a protection decorator over vulnerable external service dependencies, specifying a signature-matched local rescue fallback:
    ```java
    @CircuitBreaker(name = "userBreaker", fallbackMethod = "getUserFallback")
    public String getUser(Long id) {
        return restTemplate.getForObject("http://USER-SERVICE/users/" + id, String.class);
    }
    public String getUserFallback(Long id, Throwable t) {
        return "{\"id\":" + id + ", \"name\": \"Fallback User\"}";
    }
    ```
