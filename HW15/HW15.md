## 1. JDBC vs Hibernate?
 
JDBC is the low-level Java API for connecting to databases, where you manually write SQL, execute it, and map the ResultSet to Java objects yourself. Hibernate is an ORM framework built on top of JDBC that automates all of that — you define entities with annotations and Hibernate generates the SQL, executes it, and maps results back to objects automatically. JDBC gives you full control and is good for complex custom queries, but requires a lot of boilerplate. Hibernate reduces boilerplate dramatically but adds abstraction that can sometimes generate inefficient SQL. In Spring Boot, Hibernate is the default JPA implementation and handles most database operations, with the option to fall back to native SQL when needed.
 
## 2. Statement vs PreparedStatement vs CallableStatement?
 
Statement sends raw SQL to the database every time and recompiles it each execution — it is simple but vulnerable to SQL injection and slow for repeated queries. PreparedStatement uses a parameterized template that is compiled once and reused — parameters are filled in separately so they are treated as data not SQL, which prevents SQL injection and improves performance. CallableStatement is used to execute stored procedures in the database — it extends PreparedStatement and supports IN, OUT, and INOUT parameters. In practice, always use PreparedStatement over Statement for any query with user input, and use CallableStatement only when you need to call a stored procedure.
 
## 3. How to prevent SQL injection?
 
The primary way is to use PreparedStatement with parameterized queries instead of concatenating user input directly into SQL strings. In JPA and Spring Data, this is handled automatically because queries use named parameters or method name conventions. Additional measures include input validation to reject unexpected characters, limiting database user permissions so even if injection occurs the damage is limited, and using an ORM like Hibernate which generates parameterized SQL by default. Never concatenate user input directly into a SQL string.
 
## 4. What is ORM?
 
ORM stands for Object-Relational Mapping. It bridges the gap between Java objects and relational database tables by automatically mapping classes to tables, fields to columns, and object instances to rows. Instead of writing SQL and manually mapping ResultSets, you annotate your class with @Entity and @Id, extend JpaRepository, and the framework generates queries for you. The trade-off is that complex queries may not be as optimized as hand-written SQL, but for standard CRUD operations ORM dramatically reduces boilerplate code.
 
## 5. JPA vs Hibernate?
 
JPA is the Java Persistence API — it is a specification, meaning it defines the standard interfaces and annotations like @Entity, @Id, and @OneToMany, but it does not provide any implementation. Hibernate is the most popular implementation of JPA — it provides the actual engine that processes those annotations and generates SQL. The relationship is like an interface versus its implementation class. You code against JPA interfaces, and Hibernate does the work underneath. Spring Boot uses Hibernate as the default JPA provider, which is why you see both JPA annotations and Hibernate behavior in a Spring Boot project.
 
## 6. What are the Persistent States in Entity Lifecycle?
 
An entity goes through four states. Transient means the object has been created with new but is not associated with any persistence context — the database doesn't know about it yet. Persistent means the entity is associated with an active persistence context and any changes to it are automatically tracked and synchronized to the database. Detached means the entity was previously persistent but the persistence context has been closed — changes are no longer tracked. Removed means the entity is scheduled for deletion from the database on the next flush. Understanding these states is important because calling save() on a transient object does INSERT, while saving a persistent object does UPDATE.
 
## 7. Mapping relationships?
 
JPA supports four relationship types. @OneToOne maps a one-to-one relationship, like a user and their profile. @OneToMany maps one entity to multiple related entities, like a department with many employees. @ManyToOne is the inverse side of OneToMany, like an employee belonging to one department. @ManyToMany maps a many-to-many relationship, like students and courses, which requires a junction table. The owning side of the relationship holds the foreign key and is responsible for persisting the relationship, while the inverse side uses the mappedBy attribute.
 
## 8. What is cascade type?
 
Cascade type defines what database operations are automatically propagated from a parent entity to its related child entities. CascadeType.PERSIST means when you save the parent, the children are saved too. CascadeType.REMOVE means when you delete the parent, the children are deleted too. CascadeType.MERGE propagates merge operations. CascadeType.ALL applies all cascade types. The most common in practice is CascadeType.ALL for tightly coupled parent-child relationships like an order and its order items. Be careful with CascadeType.REMOVE on shared entities — for example cascading delete from a department would delete all its employees.
 
## 9. What is fetch type?
 
Fetch type controls when related entities are loaded from the database. EAGER means the related entity is loaded immediately together with the parent entity in the same query. LAZY means the related entity is only loaded when you actually access it, using a separate query at that point. The default is EAGER for @ManyToOne and @OneToOne, and LAZY for @OneToMany and @ManyToMany. In practice LAZY is preferred for collections because EAGER loading of large collections can cause performance issues and the N+1 query problem. You load the data only when needed.
 
## 10. What is first level cache and second level cache?
 
First level cache is the persistence context itself, scoped to a single transaction or session. When you load an entity within the same session, Hibernate returns the cached object instead of hitting the database again — this is automatic and always enabled. Second level cache is optional and shared across sessions and transactions. It caches entity data at the SessionFactory level so multiple sessions can benefit from it. You configure it explicitly using providers like Ehcache or Redis, and annotate entities with @Cache. Use second level cache for read-heavy, rarely-changing data like reference data or configuration. The trade-off is cache invalidation complexity when data changes.
 
## 11. Left join vs right join vs inner join vs outer join vs cross join?
 
Inner join returns only rows that have matching records in both tables — unmatched rows from either side are excluded. Left join returns all rows from the left table and the matched rows from the right table — unmatched right side rows appear as NULL. Right join is the mirror of left join — all rows from the right table, NULL for unmatched left side. Full outer join returns all rows from both tables, with NULL where there is no match on either side. Cross join returns the Cartesian product of both tables — every row from the left combined with every row from the right, no join condition needed. In practice inner join and left join are used most frequently. Right join can always be rewritten as a left join by swapping the table order.
 
## 12. Union vs Union ALL?
 
Both combine the result sets of two SELECT queries with the same column structure. The difference is that UNION removes duplicate rows from the combined result, while UNION ALL keeps all rows including duplicates. UNION ALL is faster because it skips the deduplication step. Use UNION when you need unique results across both queries, and UNION ALL when you know results won't overlap or you want all rows including duplicates and performance matters. In most cases UNION ALL is preferred unless duplicate removal is specifically required.
