## 1. equals vs ==?
 
The key difference is that == compares references while equals compares content. == checks whether two variables point to the same object in memory. equals by default also compares references, but most classes like String, Integer, and custom classes override it to compare the actual content. So two String objects with the same value return false with == but true with equals. The rule is: use == for primitives and reference identity checks, use equals for content comparison.
 
## 2. How to inject a bean with the same type?
 
When multiple beans of the same type exist, Spring first tries by type, then falls back to by name using the variable name. If that also fails, it throws NoUniqueBeanDefinitionException. To resolve this, use @Primary on one bean definition to mark it as the global default, so Spring picks it automatically when there is ambiguity. Or use @Qualifier at the injection point to specify exactly which bean you want by name. @Primary is a global default while @Qualifier gives precise control at each individual injection site.
 
## 3. OOP 4 principles?
 
The four principles of OOP are Encapsulation, Inheritance, Polymorphism, and Abstraction. Encapsulation means hiding internal state and exposing only necessary behavior — achieved through access modifiers like private, protected, and public, combined with getter and setter methods to control access to fields. Inheritance means a subclass extends a parent class using the extends keyword and reuses its fields and methods, reducing code duplication — a class can also implement multiple interfaces using the implements keyword. Polymorphism has two forms: runtime polymorphism through method overriding, where a subclass provides its own implementation of a parent method and the correct version is resolved at runtime based on the actual object type; and compile-time polymorphism through method overloading, where multiple methods share the same name but differ in signature, resolved at compile time. Abstraction means hiding implementation details and exposing only the essential contract — achieved through abstract classes, which can have both abstract and concrete methods but cannot be instantiated, and interfaces, which define a pure contract that implementing classes must fulfill, supporting multiple implementation in Java.
 
## 4. What is the Executor library?
 
The Executor framework manages thread pools so you don't have to manually create and destroy threads. The core interface is ExecutorService, and the main implementation is ThreadPoolExecutor where you configure core pool size, max pool size, keep alive time, and the task queue. When a task is submitted, if core threads are available it runs immediately, otherwise it goes into the queue, and if the queue is full a new thread is created up to the max pool size — beyond that a rejection policy is triggered. In Spring Boot I use ThreadPoolTaskExecutor with @Async, configuring separate pools for primary tasks like order processing and auxiliary tasks like email notifications so they never compete for the same resources.
 
## 5. Java 21 features?
 
Java 21 is an LTS release with several major features. Virtual Threads allow the JVM to manage millions of lightweight threads with minimal overhead, dramatically improving concurrency without changing application code. Pattern matching for switch allows type-based case handling directly in switch statements without manual casting. Record patterns enable deconstruction of record types in pattern matching. Sequenced Collections adds a unified interface for ordered collections with methods like getFirst() and getLast(). Virtual threads are the most impactful feature — combined with modernized Tomcat, they reduce the need for reactive programming with WebFlux in many high-concurrency scenarios.
 
## 6. What is ThreadLocal?
 
ThreadLocal provides each thread with its own independent copy of a variable so threads do not interfere with each other. The most common use case is storing the current logged-in user per request thread — you set it at the start of the request and any code in that thread can access it without passing it as a parameter. The critical rule is to always call remove() at the end of the request because web applications use thread pools and threads are reused. Failing to clean up causes the next request on the same thread to see the previous user's data, which is a data leak.
 
## 7. Java versions?
 
The major LTS versions are Java 8, 11, 17, and 21. Java 8 introduced Lambda, Stream API, Optional, and the new Date/Time API. Java 11 added the standardized HTTP Client, ZGC, and Epsilon GC. Java 17 introduced Records, Sealed classes, Switch expressions, and Text blocks — it is also the minimum version required by Spring Boot 3. Java 21 introduced Virtual Threads, Pattern matching for switch, and Record patterns. In my project I used Spring Boot 3 with Java 17, and I understand the javax to jakarta package rename is the most significant breaking change when upgrading from Spring Boot 2 to 3.
 
## 8. How do you validate input data in Spring Boot?
 
I add constraint annotations to the fields in the DTO class, such as @NotNull to reject null values, @NotBlank for empty strings, @Email for email format, @Min and @Max for number ranges, and @Pattern for custom regular expressions. Then in the controller I add @Valid to the @RequestBody parameter to trigger automatic validation before the request reaches the business logic. If any constraint is violated, Spring throws a MethodArgumentNotValidException. I catch that in my @RestControllerAdvice global exception handler and return a 400 Bad Request response with a descriptive error message so the client knows exactly what went wrong.
 
## 9. Injection types?
 
There are three types of dependency injection in Spring. Constructor injection is the recommended approach — all dependencies are required at object creation time, which guarantees the object is always in a valid state and prevents NullPointerException at runtime. It also makes unit testing easy because you can pass mock objects directly through the constructor. Field injection with @Autowired directly on the field is convenient but the field is private, making it hard to inject mocks without reflection. Setter injection makes dependencies optional which can lead to incomplete object states, so it is rarely used today. If circular dependencies occur with constructor injection, use @Lazy to defer creation of one bean.
 
## 10. Have you used PATCH?
 
PATCH is used for partial updates, unlike PUT which replaces the entire resource. I have used it when updating only specific fields of a resource without sending the full object. The important distinction is that PATCH is not guaranteed to be idempotent — if the operation sets a field to a specific value it is idempotent, but if it performs a relative change like incrementing a counter it is not. This is different from PUT which is always idempotent because it always replaces the entire resource with the same data.
 
## 11. What do you analyze for GC when you use JProfiler?
 
When analyzing GC with JProfiler, I look at two main things: pattern and pause. For pattern, I check how frequently GC is occurring and what type — minor GC in young generation should be frequent and fast, while major or full GC in old generation should be infrequent. If full GC is happening too often it indicates memory pressure. For pause, I check how long each GC pause lasts — long Stop-the-World pauses cause latency spikes in the application. I also look at heap usage over time to detect memory leaks, which show as a continuously rising heap that never drops after GC. If heap usage keeps growing without being collected, it means objects are being referenced longer than expected or there is a genuine leak.
 
## 12. Garbage collector?
 
Java's garbage collector automatically reclaims heap memory from objects that are no longer reachable. The heap is divided into young generation and old generation. New objects go into young generation, and minor GC runs frequently to collect short-lived objects. Objects that survive enough minor GCs are promoted to old generation, which is collected less frequently by major GC. Full GC collects both generations and causes the longest pauses. Java has several GC implementations: Serial GC for single-threaded environments, Parallel GC for throughput, G1 GC which is the default since Java 9 and balances throughput with pause time by dividing heap into regions, ZGC introduced in Java 11 as a low-latency collector that keeps pauses under a millisecond even on large heaps, and Epsilon GC which is a no-op collector for benchmarking. To tune GC you adjust heap size with -Xms and -Xmx, young generation size with -Xmn, and choose the appropriate collector based on your latency vs throughput requirements.
