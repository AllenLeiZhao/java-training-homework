## 1. Unit Test and Integration Test for My Project

I wrote both unit tests and integration tests for my **employee-department** project (https://github.com/AllenLeiZhao/employee-department). The test directory mirrors the source directory one-to-one:

```
src/main/java/.../employeedepartment          src/test/java/.../employeedepartment
├── config/NameAggregationConfig.java         ├── config/NameAggregationConfigTest.java
├── controller/EmployeeController.java        ├── controller/EmployeeControllerTest.java
├── controller/NameAggregationController.java ├── controller/NameAggregationControllerTest.java
├── exception/NameAggregationException...     ├── exception/NameAggregationExceptionTest.java, ...HandlerTest.java
├── mapper/ (4 mappers)                       ├── mapper/ (4 mapper tests)
├── service/EmployeeServiceImp.java           ├── service/EmployeeServiceImpTest.java
├── service/NameAggregationServiceImpl.java   ├── service/NameAggregationServiceImplTest.java
                                              ├── EmployeeDepartmentApplicationTests.java (context loads)
                                              └── EmployeeDepartmentIntegrationTest.java (integration)
```

**Unit tests (fast, isolated, no Spring context or a thin slice of it):**
- **Service layer** — plain JUnit 5 + Mockito. Repositories are mocked with `@Mock`, the service is instantiated by hand, and every method plus every failure branch (employee/department/address not found) is asserted.
- **Web layer** — `@WebMvcTest` loads only the controller slice; `EmployeeService` is replaced by `@MockitoBean`, and MockMvc verifies status codes, JSON bodies, and Bean Validation (blank name → 400).
- **Resilience logic** — `NameAggregationServiceImpl` is tested with a mocked `RestTemplate` but a **real** Resilience4j Retry and CircuitBreaker: success path, retry-then-downgrade, circuit-open short-circuit (downstream never called), invalid downstream response, and the recovery-file persistence including its IO failure branch (`@TempDir`).

**Integration tests (real Spring context, real layers wired together):**
- `@SpringBootTest + @AutoConfigureMockMvc` against an **in-memory H2 database** (configured in `src/test/resources/application.properties`), so CI needs no external PostgreSQL.
- Full CRUD flow (create → get → list → update → delete), one-to-one profile flow, one-to-many address flow, many-to-many department assign/remove, and the async name-aggregation endpoint through the real service stack with only the outbound `RestTemplate` mocked.

The integration tests immediately paid off: they caught a real bug — `Employee.employeeProfile` (`@OneToOne`) had no cascade, so `PUT /api/employees/{id}/profile` failed with `TransientPropertyValueException`. Fixed by adding `cascade = CascadeType.ALL`.

**Result: 66 tests, instruction coverage 96.5%, line coverage 100%** (JaCoCo, Lombok-generated code excluded via `lombok.addLombokGeneratedAnnotation`).

## 2. TDD vs BDD vs DDD

They answer different questions: TDD = *how do I write code*, BDD = *how do I describe behavior*, DDD = *how do I model the business*.

| | TDD (Test-Driven Development) | BDD (Behavior-Driven Development) | DDD (Domain-Driven Design) |
|---|---|---|---|
| Level | Code / developer practice | Requirement / collaboration practice | Architecture / modeling philosophy |
| Cycle | **Red → Green → Refactor**: write a failing test first, write the minimum code to pass, then refactor | Write executable specifications in natural language: **Given–When–Then** | Model the business domain: Entity, Value Object, Aggregate, Repository, Domain Event, **Bounded Context**, Ubiquitous Language |
| Audience | Developers | Developers + QA + Product (shared language) | Developers + domain experts |
| Typical tools | JUnit, Mockito | Cucumber, JBehave, Spock | No tool — patterns (often with Spring, Axon) |

- **TDD** gives you a safety net and forces testable design; the test suite becomes living documentation of the code.
- **BDD** is TDD "shifted left" to requirements: `Given an employee exists, When I DELETE /api/employees/1, Then the response message says it was deleted` — product people can read and validate it.
- **DDD** is about carving a large system into bounded contexts (e.g. Employee context vs Payroll context) so each microservice owns one domain model. It has nothing to do with testing per se, but it makes the codebase testable by keeping domain logic out of controllers.

## 3. What is JUnit

JUnit is the de-facto standard **unit testing framework for Java**. Current major version is **JUnit 5** (a.k.a. Jupiter), composed of Platform (the launcher build tools call), Jupiter (the new API), and Vintage (runs old JUnit 4 tests).

Key annotations I use:

| Annotation | Purpose |
|---|---|
| `@Test` | Marks a test method |
| `@BeforeEach` / `@AfterEach` | Setup / teardown around every test |
| `@BeforeAll` / `@AfterAll` | Once per class (static) |
| `@DisplayName` | Readable test name |
| `@ParameterizedTest` + `@ValueSource`/`@CsvSource` | Same test, many inputs |
| `@Disabled` | Skip a test |
| `@ExtendWith(...)` | Plug in extensions, e.g. `MockitoExtension`, `SpringExtension` |

Assertions: `assertEquals`, `assertThrows`, `assertAll`, plus I usually add **AssertJ** (`assertThat(x).isEqualTo(...)`) for fluent, readable assertions. Maven Surefire picks tests up automatically during `mvn test`.

## 4. What is Mockito

Mockito is the most popular **mocking framework** for Java. In a unit test I want to test one class in isolation, so its collaborators (repository, RestTemplate, another service) are replaced with **mocks** — fake objects whose behavior I script.

Core API:

- `@Mock` / `mock(Class)` — create a fake dependency.
- `when(repo.findById(1L)).thenReturn(Optional.of(employee))` — **stub** behavior.
- `thenThrow(...)`, `thenAnswer(...)` — simulate failures or dynamic answers.
- `verify(repo).save(any(Employee.class))`, `verify(repo, never()).save(any())` — assert interactions happened (or not).
- Argument matchers `any()`, `eq()`, `anyLong()`; `@ExtendWith(MockitoExtension.class)` wires everything up.

In Spring Boot tests, `@MockitoBean` (Boot 3.4+, replaces the deprecated `@MockBean`) drops a Mockito mock into the application context — that is how I mock `EmployeeService` inside `@WebMvcTest`.

Mockito cannot easily mock static/final/constructor calls (there is `mockStatic`, but needing it is usually a design smell).

## 5. How Do You Test Your Application

I follow the **test pyramid** — many fast unit tests at the bottom, fewer integration tests, few end-to-end tests:

1. **Unit tests (majority)** — JUnit 5 + Mockito, no Spring context. Service methods with mocked repositories; every branch including exception paths.
2. **Web-slice tests** — `@WebMvcTest` + MockMvc: HTTP status, JSON shape, validation errors, exception-handler output. Only the MVC layer loads, so they run in milliseconds.
3. **Data-slice tests** — `@DataJpaTest` with H2 when there are custom queries to verify.
4. **Integration tests** — `@SpringBootTest` with the real context and an in-memory H2; exercise complete flows through controller → service → repository → DB. External HTTP dependencies are mocked at the boundary (`@MockitoBean RestTemplate`).
5. **Manual/API tests** — Postman collection against the running app; the Jenkins pipeline also does a `curl` smoke test after deploying to EC2.
6. **Coverage gate** — JaCoCo `check` fails the build below 90%; the report feeds SonarQube so coverage is visible per merge.

Everything runs on every push via the CI pipeline (`mvn clean verify`), so a red test blocks the Docker build and deployment.

## 6. What Tools Do You Use to Do Code Quality Analysis

- **SonarQube** — the main one: static analysis for bugs, code smells, vulnerabilities, security hotspots, duplication, and coverage; enforced via a **Quality Gate** in the CI pipeline (see section 12).
- **JaCoCo** — test coverage measurement; produces XML consumed by SonarQube and enforces the 90% minimum with `jacoco:check`.
- **IDE inspections** — IntelliJ IDEA built-in inspections plus the SonarQube for IDE (SonarLint) plugin, which catches the same rules while I type.
- **Compiler + Maven** — `-Werror`-style discipline: treat warnings seriously, Maven Enforcer for dependency convergence.
- Other common tools in the ecosystem: **Checkstyle** (formatting/style), **PMD** (bad practices), **SpotBugs** (bytecode bug patterns), **OWASP Dependency-Check / Snyk** (vulnerable dependencies). SonarQube covers most of these categories in one place, which is why teams standardize on it.
- **Code review** — tooling never replaces a human PR review; Sonar just removes the mechanical part of it.

## 7. Authentication vs Authorization

- **Authentication (AuthN) — "Who are you?"** Verifying identity: username/password, JWT, OAuth2 token, biometrics, MFA. Fails with **401 Unauthorized**.
- **Authorization (AuthZ) — "What are you allowed to do?"** Checking permissions of an already-identified user: roles, scopes, ACLs. Fails with **403 Forbidden**.

Authentication always happens first; authorization is meaningless without a verified identity. Example: I log into the HR system with my password (authentication succeeds), but when I try to open salary administration the system rejects me because I am not an HR admin (authorization fails).

In Spring Security: authentication is handled by the filter chain / `AuthenticationManager`; authorization by `authorizeHttpRequests().requestMatchers("/admin/**").hasRole("ADMIN")` or method-level `@PreAuthorize("hasRole('ADMIN')")`.

## 8. Encryption vs Hashing vs Encoding

| | Encoding | Encryption | Hashing |
|---|---|---|---|
| Purpose | **Data format transformation** for transport/storage | **Confidentiality** — hide content | **Integrity / verification** — fingerprint |
| Reversible? | Yes, by anyone (no secret) | Yes, but **only with the key** | **No** — one-way |
| Secret involved | None | Key (symmetric or key pair) | None (salt is not a secret key) |
| Examples | Base64, URL encoding, UTF-8 | AES (symmetric), RSA (asymmetric), TLS | SHA-256, bcrypt, SHA-1/MD5 (broken) |
| Typical use | JWT payload, email attachments | HTTPS traffic, encrypted DB columns | Password storage, file checksums, digital signatures |

Key points:
- **Encoding is NOT security.** Base64 "protects" nothing — a JWT payload is only Base64-encoded, which is why you never put secrets in it.
- **Encryption** protects data in transit/at rest and is meant to be reversed by the legitimate key holder. Symmetric (AES) = same key both sides, fast; asymmetric (RSA) = public key encrypts, private key decrypts.
- **Hashing** is for when you never need the original back. Passwords must be hashed (never encrypted!) with a **slow, salted** algorithm — bcrypt/argon2 — so leaked databases can't be brute-forced with rainbow tables.

## 9. How Do You Secure Your Application

Defense in depth — multiple layers:

1. **Transport** — HTTPS/TLS everywhere; no credentials or tokens over plain HTTP.
2. **Authentication & Authorization** — Spring Security filter chain; stateless JWT for APIs; role-based access (`@PreAuthorize`); short token lifetimes with refresh tokens.
3. **Password storage** — bcrypt via `BCryptPasswordEncoder`, never plaintext or reversible encryption.
4. **Input validation** — Bean Validation (`@Valid`, `@NotBlank`, `@Email` — as in my `EmployeeDto`); reject bad input at the edge with 400.
5. **Injection protection** — JPA/Hibernate parameterized queries (never string-concatenated SQL); escape output to prevent XSS; CSRF protection for browser sessions.
6. **Secrets management** — no credentials in Git; environment variables / Jenkins Credentials (my pipeline injects Docker Hub and SSH credentials via `withCredentials`/`sshagent`), AWS IAM roles instead of hard-coded keys.
7. **Dependency hygiene** — keep Spring Boot patched; scan dependencies (OWASP Dependency-Check/Snyk); SonarQube security hotspots in CI.
8. **Least privilege** — DB user with only needed grants; EC2 security groups open only required ports; containers don't run as root.
9. **Error handling & logging** — global exception handlers return sanitized messages (no stack traces to clients); audit logs for sensitive operations; rate limiting/throttling at the gateway.

## 10. What is JWT

**JWT (JSON Web Token)** is a compact, self-contained, digitally signed token used for stateless authentication. Structure — three Base64Url parts separated by dots:

```
header.payload.signature
```

- **Header** — algorithm and type: `{"alg":"HS256","typ":"JWT"}`
- **Payload** — claims: `sub` (user), `iat` (issued at), `exp` (expiry), plus custom claims like roles.
- **Signature** — `HMACSHA256(base64(header) + "." + base64(payload), secret)` — or RS256 with a private/public key pair.

**Flow:** login with credentials → server verifies and returns a signed JWT → client stores it and sends `Authorization: Bearer <token>` on every request → server validates the **signature and expiry** and trusts the claims — no session lookup, no server-side state.

**Why it's popular:** stateless (scales horizontally, perfect for microservices — every service can verify the token with the shared secret/public key), cross-domain friendly, carries its own claims.

**Caveats:** payload is only *encoded*, not encrypted — never put secrets in it; tokens can't be revoked before expiry (mitigate with short TTL + refresh tokens or a blacklist); always transmit over HTTPS; keep the signing secret safe.

## 11. What is OAuth2

**OAuth2 is an authorization framework** — it lets a user grant a third-party application **limited access** to their resources **without sharing their password**. Classic example: "Sign in with Google" or letting an app read your GitHub repos.

Four roles:
- **Resource Owner** — the user.
- **Client** — the third-party app requesting access.
- **Authorization Server** — issues tokens (Google/GitHub/Keycloak/Okta).
- **Resource Server** — the API holding the data, accepts access tokens.

Main flow (**Authorization Code**, the recommended one, today with **PKCE**):
1. Client redirects the user to the authorization server with `client_id`, `redirect_uri`, `scope`.
2. User logs in **at the authorization server** and consents.
3. Authorization server redirects back with a one-time **authorization code**.
4. Client exchanges the code (+ client secret / PKCE verifier) for an **access token** (and optionally a **refresh token**) via a back-channel call.
5. Client calls the resource server with `Authorization: Bearer <access_token>`; scopes limit what it can do.

Other grant types: **Client Credentials** (service-to-service, no user), **Refresh Token** (renew access without re-login); the old Implicit and Password grants are deprecated.

**OAuth2 vs JWT:** OAuth2 is the *protocol* deciding how tokens are obtained; JWT is a *token format* — access tokens are often (not necessarily) JWTs. **OIDC (OpenID Connect)** is an identity layer on top of OAuth2 that adds the `id_token` (always a JWT) for authentication — that's what "Login with Google" actually uses.

## 12. CI/CD Pipeline Integration — JaCoCo + SonarQube (employee-department)

Done in https://github.com/AllenLeiZhao/employee-department.

### 12.1 JaCoCo integration (pom.xml)

`jacoco-maven-plugin` with three executions:
- **prepare-agent** — attaches the JaCoCo agent to the test JVM;
- **report** — generates HTML + XML (`target/site/jacoco/jacoco.xml`) after `test`;
- **check** — bound to `verify`, fails the build if **instruction or line coverage < 90%**.

```xml
<limit><counter>INSTRUCTION</counter><value>COVEREDRATIO</value><minimum>0.90</minimum></limit>
<limit><counter>LINE</counter><value>COVEREDRATIO</value><minimum>0.90</minimum></limit>
```

Supporting choices: the bootstrap class `EmployeeDepartmentApplication` is excluded; `lombok.config` sets `lombok.addLombokGeneratedAnnotation = true` so Lombok-generated getters/setters/equals don't distort coverage; H2 (test scope) lets the integration tests run in CI without PostgreSQL.

**Current result: 66 tests — instruction coverage 96.5%, line coverage 100% — gate passed.**

### 12.2 SonarQube in the Jenkins pipeline

`sonar-maven-plugin` added to the pom with properties (`sonar.projectKey=employee-department`, `sonar.coverage.jacoco.xmlReportPaths` pointing at the JaCoCo XML so Sonar shows the same coverage). New Jenkinsfile stages, before the Docker build:

```groovy
stage('Unit Test & Coverage') {
    steps { sh './mvnw clean verify' }              // JUnit 5 + JaCoCo report + 90% gate
    post { always { junit 'target/surefire-reports/*.xml' } }
}
stage('SonarQube Analysis') {
    steps {
        withSonarQubeEnv('sonarqube') {             // server configured in Manage Jenkins
            sh './mvnw -q sonar:sonar'
        }
    }
}
stage('Quality Gate') {
    steps {
        timeout(time: 10, unit: 'MINUTES') {
            waitForQualityGate abortPipeline: true  // SonarQube webhook decides pass/fail
        }
    }
}
```

Full pipeline order: **Checkout → Unit Test & Coverage → SonarQube Analysis → Quality Gate → Build Jar → Build Docker Image → Push Docker Image → Deploy to EC2 → Verify (curl smoke test)**.

One-time Jenkins/SonarQube setup this relies on:
1. Run SonarQube (e.g. `docker run -d -p 9000:9000 sonarqube:community`).
2. Jenkins: install *SonarQube Scanner* plugin; add the server + auth token under **Manage Jenkins → System → SonarQube servers** with name `sonarqube`.
3. SonarQube: create a webhook pointing to `http://<jenkins>/sonarqube-webhook/` so `waitForQualityGate` gets the result.

Effect: any push that breaks a test, drops coverage below 90%, or fails the Sonar quality gate **never reaches Docker build or EC2 deployment**.
