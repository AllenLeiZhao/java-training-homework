### Architecture: two logins, one stateless session

```
                    ┌──────────────────────────────── Angular (localhost:4200) ─┐
 username/password ─┤ POST /auth/token  (HTTP Basic)          ──────────────┐   │
                    │                                                       ▼   │
 "Sign in with      │ redirect /oauth2/authorization/google → Google        JWT │
  Google"          ─┤ … Spring finishes OIDC, session cookie set,       in local│
                    │ redirect → /login/oauth2/success                  storage │
                    │ POST /auth/token (withCredentials: session)  ─────────┘   │
                    └───────────── every API call: Authorization: Bearer <jwt> ─┘
```

The design insight that kept the backend change tiny: **`/auth/token` was already guarded by `anyRequest().authenticated()`** — it never cared *how* you authenticated. Basic credentials and a Google-established session cookie both work, so the SPA funnels both login paths into the same JWT and stays stateless afterwards. (Cookie note: `localhost:4200 → localhost:8082` is *same-site* — SameSite ignores ports — so the session cookie rides along on the token exchange with `withCredentials: true`.)

### Backend changes (3 small pieces)

| Change | Why |
|---|---|

### CI/CD — the SPA ships inside the jar

Same-origin deployment: the Angular build is packaged into Spring Boot's `classpath:/static`, so EC2 exposes exactly one origin and `http://<EC2>:8080/login` serves the Angular login page.

| Piece | What it does |

## 1. What is HTML and what is its main responsibility in a web page?

HTML (HyperText Markup Language) is the document language of the web. Its one job is **structure and meaning**: it says "this is a heading, this is a paragraph, this is a table of employees, this is a form field" — the *what*, never the *how it looks* (CSS) or *how it behaves* (JavaScript). In my Angular app the templates are still just HTML at heart: the employee table is a `<table>`, the login form is a `<form>` with `<input>`s; Angular only decides *when* to render them.

## 2. What is an HTML file essentially made of?

Nested **elements** — text wrapped in tags, with attributes, forming a tree. One root (`<html>`), metadata (`<head>`), visible content (`<body>`), and inside that a hierarchy of containers and text nodes. Everything else on a page — styling, scripts, images — is *referenced from* this tree.

## 3. What does `<!DOCTYPE html>` do and why is it required?

It tells the browser "parse this as modern, standards-mode HTML." Without it, browsers fall back to **quirks mode** — a bug-compatibility mode that emulates 1990s layout behavior (e.g., a different box model), because old sites depended on those bugs. It isn't a tag; it's a parser instruction, and it must be the first thing in the file.

## 4. What is an HTML tag and how does it work?

A tag is the bracket syntax that marks where an element starts and ends: `<td>Allen</td>`. The browser's parser reads the opening tag, creates a corresponding node in memory, gives it the attributes, then everything until the closing tag becomes that node's children. Some elements are self-contained ("void") and have no closing tag — `<input>`, `<img>`, `<br>`.

## 5. What is the difference between the `<head>` and `<body>` sections?

`<head>` is metadata *about* the document — title, charset, stylesheet links, scripts, favicon. Nothing in it renders on the page. `<body>` is everything the user actually sees and interacts with. Rule of thumb: `<head>` talks to the browser, `<body>` talks to the human.

## 6. What is an HTML attribute and why is it needed?

An attribute is a name/value pair on the opening tag that configures the element: `<input type="password" id="password" required>`. The tag says *what* the element is; attributes say *which variant, pointing where, behaving how* — `href` on a link, `src` on an image, `type` on an input. My login form works as a password field purely because of `type="password"`.

## 7. How does the browser parse HTML and convert it into the DOM?

The browser streams the bytes, decodes them into characters, **tokenizes** them into tags/attributes/text, and builds a tree of live objects — the DOM (Document Object Model). Each element becomes a node with parent/children pointers. The key mental shift: the DOM is not the HTML file — it's the in-memory data structure built *from* it, and it keeps living afterwards: JavaScript can add and remove nodes, and the page re-renders. That's exactly what Angular does all day — my employee table rows are DOM nodes Angular creates from data, they never existed in any HTML file.

## 8. What is CSS and what problem does it solve?

CSS (Cascading Style Sheets) describes presentation: colors, spacing, fonts, layout. The problem it solves is **separating looks from structure** — before CSS, styling lived inside HTML (`<font>` tags, layout tables), so restyling a site meant rewriting every page. With CSS, one stylesheet rules the whole app: my `styles.css` defines `.card`, `.button`, `.error` once and every component uses them.

## 9. Why is CSS not considered a programming language?

It's **declarative, not imperative**: you state rules ("all `.error` text is red"), and the browser figures out how to apply them. There's no control flow you author — no loops, no functions with logic, no state you mutate step by step. Modern CSS has variables and `calc()`, but you still can't express an algorithm in it. That's not a weakness — being non-Turing-complete is why a stylesheet can never crash or loop forever.

## 10. What are the three ways to apply CSS to an HTML page?

1. **External stylesheet** — `<link rel="stylesheet" href="styles.css">`: cacheable, shared across pages; the default choice (my `styles.css`).
2. **Internal** — a `<style>` block in `<head>`: fine for one-off pages or emails.
3. **Inline** — `style="color: red"` on the element itself: highest specificity, lowest reusability.

## 11. Why are inline styles generally not recommended?

They defeat every advantage CSS has: no reuse (copy-pasted on every element), no caching, hardest to override (they beat class selectors, so fixing them requires `!important` wars), and they scatter design decisions across the markup instead of one place. Change "brand blue" and you're grepping the whole codebase. Legitimate uses are narrow: values computed at runtime (a progress bar width) or HTML emails.

## 12. What is the difference between id selectors and class selectors in CSS?

`#login-form` targets **one** element — ids must be unique per page; `.card` targets **every** element carrying that class. Ids also have higher specificity, which sounds convenient and becomes a trap: styles anchored to ids are hard to override and impossible to reuse. My rule: classes for styling (reusable, composable), ids for JavaScript hooks, form labels (`for="password"`), and URL fragments.

## 13. What is the difference between margin and padding?

Both are whitespace, on opposite sides of the border. **Padding** is inside — between the content and its border; it gets the element's background color and grows its clickable area. **Margin** is outside — distance to the neighbors; always transparent, and vertical margins can collapse into each other. My `.card` uses `padding: 24px` so text doesn't touch the border, and the table cells use padding so rows feel airy; the space *between* cards would be margin.

## 14. What is the difference between `position: relative` and `position: absolute`?

`relative` keeps the element in normal flow — the layout still reserves its original spot — and offsets (`top/left`) nudge it visually from where it *would* have been. `absolute` removes it from flow entirely and positions it against **the nearest positioned ancestor** (anything with `position` other than `static`). The two work as a pair: parent `relative` (as the anchor, unmoved), child `absolute` (pinned corner badge, dropdown, tooltip). In my rednote card templates that's exactly how the price-tag flags sit on card corners.

## 15. What is JavaScript used for in a web application?

Behavior — everything that happens *after* the page loads: reacting to clicks and input, fetching data from APIs without a reload, and updating the DOM with the results. My HW23 app is the demonstration in the extreme: the HTML file is nearly empty, and JavaScript (compiled from Angular/TypeScript) builds the login flow, calls the Spring API with a JWT, and renders the employee table from JSON.

## 16. What are the two ways to include JavaScript in an HTML file?

1. **External file** — `<script src="main.js"></script>`: cacheable, reusable; what every real app does (Angular's build emits exactly these tags).
2. **Inline** — code directly inside a `<script>…</script>` block.
Related but distinct: `defer`/`module` scripts wait for the document to be parsed before running — the modern default, replacing the old "put scripts at the bottom of `<body>`" trick.

## 17. What is a variable in JavaScript and what types of values can it store?

A named binding to a value: `let` (reassignable), `const` (binding fixed — the object it points to can still mutate), and legacy `var` (function-scoped, avoid). JavaScript is dynamically typed, so a variable can hold any type: primitives (string, number, boolean, `null`, `undefined`, bigint, symbol) or objects (plain objects, arrays, functions, dates…). The type lives with the *value*, not the variable — which is exactly why my Angular code is written in TypeScript, putting compile-time types on top.

## 18. How does JavaScript interact with HTML elements through the DOM?

The DOM is exposed as a JavaScript API. You **select** nodes (`document.querySelector('#password')`), **read/update** them (`el.textContent = '…'`, `el.classList.add('error')`), **create/remove** them (`document.createElement('tr')`, `el.remove()`), and **listen** to them (`el.addEventListener('click', …)`). Every UI framework compiles down to exactly these calls — Angular's `@for` loop over my employees is, at runtime, `createElement('tr')` and `appendChild` per row.

## 19. What are loops used for in JavaScript, and can you give an example use case?

Repeating work over a collection or until a condition flips: `for`, `for…of`, `while`, plus array methods (`map`, `filter`, `forEach`) which are loops in functional clothing. The canonical front-end use case is **rendering a list from API data** — take the JSON array from `GET /api/employees` and produce one table row per element. In vanilla JS that's a `for…of` appending `<tr>`s; in my Angular template it's the `@for (e of employees(); track e.id)` block.

## 20. What is an event listener and why is it important in front-end development?

A callback registered for a specific event on a specific element: `button.addEventListener('click', handler)`. It's *the* mechanism that makes pages interactive — the browser is event-driven, so all app logic starts from some event (click, input, submit, keydown, fetch completion). Listeners invert control: the browser calls *you*. Angular's `(click)="logout()"` and `(ngSubmit)="save()"` in my templates are event listeners with framework sugar; underneath it's `addEventListener` all the way down.
